## 协作规范

> **开发模式：`flexible`**（灵活模式）
>
> - `strict`（严格模式）：所有代码改动必须先更新 spec 再写代码，适合多人协作、合规要求高、需要完整追溯的项目
> - `flexible`（灵活模式）：大改动走 spec-first，小改动允许 code-first + 定期审计收敛，适合个人/小团队快速迭代
>
> 切换方式：直接修改上方 `开发模式` 的值为 `strict` 或 `flexible`；或对 AI 说"把开发模式切换为 strict/flexible"

- **先确认再修改**：修改任何 openspec 文档前，必须先告知目标文件和修改内容摘要，等确认后再动手编辑
- **自动定位**：描述需求时无需指定文件路径，AI 会根据语义自动匹配到对应的 spec 或 design 文档
- **跨文件变更说明**：若一个需求涉及多个 spec 文件，需逐一列出受影响的文件并说明各自的改动点
- **新功能和大改动必须走 spec**：先写 proposal，确认后更新 spec，再写代码；命中用户可见流程重做、状态/错误处理变化、数据契约变化、或结果受阈值影响时，不按纯小改动处理
- **小改动允许先改代码后补 spec**（仅 flexible 模式）：bug 修复、参数调整、样式微调等可以直接改代码，但功能行为有实质变化时，应在合适时机补更新对应 spec
- **审计分两类**：运行 `audit.py` 做结构审计；需要对照代码检查 spec 与实现是否偏离时，执行漂移审计
- **定期审计收敛**：完成一轮功能开发、累积 3~5 次小改动、发布前/交接前、切换会话或 AI 前，建议执行对应审计
